Response:
I pulled the repo and actually checked the data (not just the README's claims). Here's what's really in there.

**What's actually in the dataset**

- 576 press-release rows total, spanning 2026-01-02 to 2026-07-17 (~127 days with any data)
- Only **~29-30 unique tickers** have any releases at all — not the 110 the README claims are in `sources.yaml`, and nowhere near the 500 S&P 500 companies that's the stated goal
- Coverage is extremely lopsided: IBM (78 rows), Centene (53), Apollo (38), Chipotle (25), Paramount (24), NetApp (24)... down to single-digit rows for others. Most S&P 500 names have zero releases.

**Per use case — unvarnished:**

| Use case | Verdict | Why |
|---|---|---|
| **Event-driven investing** | Genuinely useful (for the ~30 covered tickers) | This is literally the stated purpose. Data is dated, sourced from official IR pages, no wire-service noise. |
| **Corporate communications analysis** | Genuinely useful | It's raw, unfiltered official corporate press releases — exactly the input this kind of analysis needs. |
| **Earnings monitoring** | Useful, real examples present | Found "to Report Q4 Results," "to Announce ... on Feb 10" type releases (~83 hits) — both the pre-announcement and the results release. |
| **M&A monitoring** | Useful, real examples present | Caught the Paramount/Warner Bros. Discovery bidding saga end-to-end and IBM's Confluent acquisition. Genuinely good signal, when the company is covered. |
| **Product launch detection** | Useful, real examples present | AMD's CES 2026 announcements, Casey's new pizza launch, etc. |
| **Supply-chain tracking** | Not really | I found essentially nothing supply-chain specific (supplier changes, plant openings, logistics). Not a strength of this dataset. |
| **Regulatory announcements** | Not really | The project explicitly excludes SEC EDGAR/8-K content by design (stated in Motivation). You won't get FDA approvals, SEC filings, or agency actions from this — only what a company chooses to put on its own IR page as a press release, which sometimes overlaps regulatory news but isn't a reliable source for it. |

**The catch that matters more than any of the above:** even where the data is "useful" in principle, coverage is so thin (30 of 500 companies, and unevenly at that) that for any of these use cases beyond a handful of names, it's not usable yet as a research dataset — it's a proof of concept. If you need event-driven signal on, say, a random S&P 500 industrial or healthcare name, there's a good chance it's simply not in there.

**Should you update the README?**

I'd hold off, and I don't think this is close. Two reasons:

1. The README is already unusually honest about being early-stage and sparse — it says so up front, twice. Adding a polished "use cases" section (event-driven investing, M&A monitoring, product launch detection, etc.) would read as a capability claim, and right now the capability is "works great for ~30 tickers, does nothing for the other 470." That's a promise the data can't back up yet.
2. The README's existing Motivation section already covers the core use case (market reaction to press releases) accurately and doesn't overclaim. Expanding the list of buzzwordy use cases before coverage improves risks becoming exactly the kind of overselling the project is implicitly positioning itself against (it takes a swipe at wire services and EDGAR for being messy/biased — don't want the README itself to start overselling).

If coverage grows toward the stated 110-500 company goal, then yes, an explicit "what this is good for" section would be worth adding — and at that point I'd write it close to what's in the table above, including the "not good for supply-chain/regulatory" caveats, since a useful README tells people what a dataset *isn't* for as much as what it is.
