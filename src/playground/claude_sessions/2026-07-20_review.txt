## Speed

Two real, unforced costs:

1. **`scrape_all.py` runs every source strictly sequentially, in-process**, with no concurrency at all — confirmed in code and in `scrape_utils.py`'s own comments. Your politeness guideline (10–30s between requests *per domain*) doesn't require this — different sources hit different domains, so running several sources concurrently (thread pool or asyncio, one worker per distinct domain) wouldn't violate it and would materially cut wall-clock time for a full run across 22+ configured sources.

2. **`scrape_q4_ir.py`'s `scrape_all_years()` relaunches a fresh Chrome instance for every year in a multi-year range** (`with sync_playwright() as p: ... browser.close()` sits inside the per-year loop in `render_news_page()`). For a single-year run this is invisible; for a `--start-year 2020 --end-year 2026` backfill it means 7 cold Chrome launches instead of 1 browser reused across 7 page navigations. That's the single biggest win available for backfill speed specifically.

## Bottom line — priority order

1. Add the 69 already-supported-platform sources to `scraper_config.yaml` (biggest coverage win, no new code).

4. Add unit tests (with saved fixture HTML) for `scrape_investorroom.py` / `scrape_notified.py` / `scrape_notified_gated.py` — they're your highest-risk, least-tested code.

6. Parallelize `scrape_all.py` across distinct domains if full-catalog run time becomes a real pain point — I'd rank this last since it's the most invasive change for the least urgent problem right now.
