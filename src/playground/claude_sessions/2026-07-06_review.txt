## Overall impression

This is a well-engineered *hobby-stage* project. The code quality is clearly above what the "early stage, sparse coverage" README modesty suggests — but the actual data coverage is genuinely thin, and there's a specific, easy-to-miss reason why: **the coverage problem is mostly a wiring problem, not a scraping problem.**

## What's actually good

- **The three-platform-scraper architecture is the right call.** Q4, InvestorRoom, and Notified/Drupal each get one generic scraper instead of one script per company. That's the correct abstraction for this problem and it's executed cleanly — shared `NewsItem`, shared `resolve_source()` precedence logic, shared CSV merge/dedup in `utils/`.
- **`detect_ir_platform.py` reuses the *exact* regexes from each scraper's docstring**, with comments pointing back to the source line. That's a nice touch that keeps detection and scraping from drifting apart.
- **`check_scraper_coverage.py` catches real config bugs** (typo'd slug, slug configured under two groups = double-scraped) — someone has already been bitten by hand-editing YAML.
- **`join_url_path()` + its test** — a small, boring fix for a real bug (missing leading slash silently produces `ir.x.comnews`), and it's the *only* function in the repo with a test. Good instinct, but tells you something (see below).
- **The "why not use BusinessWire/EDGAR" reasoning in the README is genuinely sound** — not just boilerplate justification.

## The unvarnished problems

**1. Test coverage is essentially zero.** One test file, one function (`join_url_path`). The stuff that actually breaks in a scraper — date parsing (`parse_short_date`, `date_from_url`), the CSV merge/upsert logic in `csv_utils.py`, `resolve_source_identity`'s precedence rules, the pagination-stop conditions — none of it is tested. `csv_utils.merge_into_daily_csvs` in particular has subtle logic (upsert-by-URL, "skip write if nothing changed") that's exactly the kind of code that silently corrupts data months later if it regresses.

**2. There is no automation.** No GitHub Actions, no cron, nothing. `scrape_all.py` exists and works, but nobody is running it on a schedule. Data coverage reflects this: 74 out of ~187 days in 2026 have *any* row at all, and only for the 8 sources actually wired into `config/scraper_config.yaml`. This project is currently 100% dependent on the maintainer remembering to run it.

**3. Coverage-by-config is badly lagging coverage-by-capability.** This is the big one — see below.

**4. `requires-python = ">=3.14"`** is an aggressive pin for a project that wants community contributions ("contributions welcome"). 3.14 is very new; you're narrowing your contributor pool for no clear benefit I can see in the code (I didn't spot any 3.14-only syntax).

**5. `unknown` platform is 18 of 72 sources**, including some big ones (Apple, Tesla, Nike, IBM, Qualcomm, ServiceNow). These will need real one-off investigation — no shortcut available.

## The single most important finding: your coverage gap is mostly already solved

`reports/latest/scraper_coverage.txt` says 8/72 (11%) covered. But `reports/latest/ir_platform.txt` — generated by a tool you already have — shows the *actual* platform breakdown across those 72 sources:

| Platform | Sources with that platform | Currently configured in `scraper_config.yaml` |
|---|---|---|
| notified | 32 | 3 (abbvie, teradyne, apollo) |
| q4 | 16 | 3 (costco, cdw, coinbase) |
| investorroom | 4 | 2 (chipotle, axon) |
| unknown | 18 | 0 |

That means **~54 sources already have a working generic scraper for their platform** and simply aren't in `config/scraper_config.yaml` yet. This isn't a "build a new scraper" problem for most of your gap — it's a "add an entry, run it, fix the inevitable per-site quirk" problem. Concretely:

- Notified sites: broadcom, dell, datadog, equifax, factset, honeywell, kla, micron, msci, onsemi, verizon, workday, tjx, etc. — 29 more beyond the 3 already wired.
- Q4 sites: booking, corning, doordash, expedia, flex, google, seagate, super-micro, visa, vertiv, builders-firstsource, ciena, gen-digital — 13 more.
- InvestorRoom: genuine-parts, lam-research — 2 more.

Each will need a quick manual pass (does it need `--fetch-detail-pages`? does it need `news_releases_path` or `first_page_index` overrides in `sources.yaml`, like Teradyne/Apollo already do?), but that's a couple hours per site, not a new scraper.

## Recommended next steps, in priority order

1. **Batch-onboard the 45 known-platform, unconfigured sources.** Highest ROI by far — this alone would take coverage from 11% to ~75% without writing a line of new scraping logic. Script it: for each slug with `platform != unknown`, dry-run the matching scraper, eyeball the output, add to `scraper_config.yaml` if clean.
2. **Automate the run.** A GitHub Actions workflow (daily cron) running `scrape_all.py --dry-run`... actually running it for real, committing the diff, maybe opening a PR for review rather than pushing straight to main (keeps the "links are verified before merging" guideline intact for automated contributions too). Right now the whole project stalls whenever the maintainer is busy.
3. **Add tests for the boring-but-load-bearing logic**, especially `csv_utils.merge_into_daily_csvs` (upsert correctness) and each scraper's date-parsing function. These are pure functions — cheap to test, expensive to debug once wrong data is already merged into a year of CSVs.
4. **Tackle `unknown` platforms one at a time**, starting with the ones most likely to be a variant of a platform you already handle (run `detect_ir_platform.py -v --slug X` to see which signals fired vs. didn't) before assuming each needs a bespoke scraper.
5. **Reconsider the `>=3.14` floor** if you actually want external contributors — or document why it's required, if there's a reason I didn't see.
