====== primary_wire: Tasks ======

This project uses [[https://www.pyinvoke.org/|Invoke]] as a small task runner
for repetitive maintenance commands -- regenerating the files under
''reports/latest/'', and smoke-testing the scrapers.

===== Why Invoke =====

Previously, the reports were regenerated by hand with:

  python.exe src/detect_ir_platform.py                    > reports/latest/ir_platform.csv
  python.exe src/missing_tickers.py                         > reports/latest/missing_tickers.txt
  python.exe src/check_scraper_coverage.py --write-reports

check_scraper_coverage.py writes its own two output files directly --
reports/latest/scraper_coverage_summary.txt (prose) and
reports/latest/scraper_coverage_missing.csv (CSV, header
"slug,ticker,platform,ir_url") -- rather than printing to stdout for
redirection like the other two scripts, so that both files are guaranteed
to come from the same sources.yaml/scraper_config.yaml snapshot.

That works, but it's easy to typo a path, forget one of the commands,
or forget to run it before committing. Invoke replaces this with a single
named command (''invoke reports''), documents itself (''invoke --list''),
and is plain Python underneath -- no new syntax to learn, and it runs the
same way on Windows, Mac, and Linux (a plain Makefile would not, since
''make'' is not available out of the box on Windows).

===== Setup =====

Invoke is installed as part of the ''dev'' extra:

  uv sync --extra dev

===== Usage =====

List available tasks:

  invoke --list

Regenerate all the reports (this is also the default task, so plain
''invoke'' with no arguments does the same thing):

  invoke reports

Regenerate just one report:

  invoke ir-platform
  invoke missing-tickers
  invoke scraper-coverage

''ir-platform'' and ''missing-tickers'' run their script via ''uv run
python <script>'' (using an absolute path to the script), capture stdout,
and write it to the corresponding file under ''reports/latest/''.
''scraper-coverage'' runs its script the same way but with
''--write-reports'', which makes the script write
''scraper_coverage_summary.txt'' and ''scraper_coverage_missing.csv''
directly, instead of tasks.py capturing stdout -- see that script's
docstring for why. Either way this works whether or not a virtual
environment is currently activated, and from any directory inside the repo
-- see "Running from other directories" below.

===== Smoke-testing the scrapers =====

Before pushing a change that touches a scraper or a shared utility module,
check that nothing is broken with:

  invoke smoke-test

This runs ''src/scrape_all.py --smoke-test --dry-run'', which scrapes one
randomly-picked source per distinct (scraper, extra CLI args) combination
in ''config/scraper_config.yaml'', instead of every configured source. For
example, ''cdw'' uses ''--fetch-detail-pages'' and is the only source
configured with that flag, so it always runs; ''costco'' and ''coinbase''
both use ''--fallback-to-visible'', so one of the two is picked at random
each time. This gives quick, representative coverage of every distinct code
path without waiting for a full run across every configured source, and
without hammering the same slug (and the servers behind it) on every run.
''--dry-run'' means nothing is written to ''data/''.

For a reproducible run (e.g. to attach the exact picks to a bug report):

  invoke smoke-test --seed 42

See the ''--smoke-test'' section of ''src/scrape_all.py''s docstring for
the full rationale, and ''pick_smoke_test_selection()'' in that file for
the implementation.

===== Running from other directories =====

Invoke searches upward from your current directory for ''tasks.py'', the
same way ''git'' searches upward for ''.git''. So ''invoke reports'' works
from the repo root or from any subdirectory inside the repo (e.g. ''src/'',
''data/2026/''), not just the root. It will *not* find ''tasks.py'' if you
run it from outside the repo entirely.

Script and output paths in ''tasks.py'' are resolved as absolute paths
relative to ''tasks.py'' itself (not relative to your current directory), so
regenerating reports gives the same result no matter where inside the repo
you run it from.

===== Adding new tasks =====

Tasks live in ''tasks.py'' at the repo root. Add a new ''@task''-decorated
function there, following the existing ones as a template.
